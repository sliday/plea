# PLAN.md Output Template

Use this template when generating the final PLAN.md. Fill every section. If a section has no content, write "N/A" rather than omitting it.

---

## Template

```markdown
# Plan: {short_title}

Generated by `/clarify` | {YYYY-MM-DD} | depth: {compact|standard|thorough} ({N} questions)

## Summary

{2-3 sentences: what will be built, why, and the key architectural approach}

## Decisions

| # | Question | Answer | Notes |
|---|----------|--------|-------|
| 1 | {question text} | {yes/no/value} | {brief context if answer was nuanced} |
| 2 | ... | ... | ... |

## Files

### New
- `{path}` — {what this file does}

### Modified
- `{path}` — {what changes and why}

### Deleted (if any)
- `{path}` — {why removed}

## Implementation Order

1. **{phase_name}** — {what gets built}
   - Files: {list}
   - Depends on: {none or previous phase}
   - Estimated complexity: {low/medium/high}

2. **{phase_name}** — {what gets built}
   - Files: {list}
   - Depends on: Phase 1
   - Estimated complexity: {low/medium/high}

## Non-Functional Requirements

- {requirement}: {detail} (from question #{N})
- ...

## Open Questions

- {anything unresolved from the interview}
- {items flagged as "tension" but not hard contradictions}

## Constraints

- {technical constraints discovered during interview}
- {compatibility requirements}
- {performance budgets}
```

---

## Filling Guidelines

### Summary
- First sentence: what is being built (the feature/change)
- Second sentence: the key technical approach (framework, pattern, integration point)
- Third sentence (optional): why this approach was chosen over alternatives

### Decisions Table
- Include ALL questions asked during the interview, in order
- The `Notes` column captures nuance: if the user said "yes, but only for admin users", capture that qualifier
- Pre-filled confirmations from PREFILL phase go here too, marked with source: `(confirmed from codebase)`

### Files
- Be specific about paths — use the project's actual directory structure from the SCAN phase
- For new files: describe purpose and key contents
- For modified files: describe what changes, not just "update"
- Order files by implementation phase

### Implementation Order
- Match phases to natural dependency order (schema before API before UI)
- Each phase should be independently testable
- Complexity estimate helps with sprint planning

### Non-Functional Requirements
- Only include items that emerged from the interview (rate limiting, accessibility, performance budget)
- Reference the question number that surfaced the requirement
- Be specific: "< 200ms P95 response time" not "fast"

### Open Questions
- Items the user explicitly deferred ("we'll figure that out later")
- Tensions flagged but not resolved
- Dependencies on external decisions or other team's work

### Constraints
- Hard technical constraints from the codebase (e.g., "must use existing ORM", "Node 18 minimum")
- Constraints from user answers (e.g., "no new infrastructure", "must work offline")
